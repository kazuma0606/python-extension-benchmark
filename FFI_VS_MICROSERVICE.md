# Python 高速化アーキテクチャ選択の知見

## 背景

Python のパフォーマンスボトルネックを解消するアプローチとして、大きく2つの方向性がある。

1. **FFI / Python 拡張モジュール** — 同一プロセス内で他言語の共有ライブラリを呼ぶ
2. **マイクロサービスアーキテクチャ** — 他言語で書いたサービスをプロセス間通信で呼ぶ

どちらが適切かは、要件によって明確に異なる。

---

## FFI / 拡張モジュールが有効なケース

### 適合条件

- **既存の Python コードベース**があり、特定のホットスポットだけ高速化したい
- **レイテンシが極めてシビア**（マイクロ秒単位）でプロセス間通信のオーバーヘッドすら許容できない
- **データの受け渡し量が膨大**でシリアライズコストを払いたくない
- NumPy 配列を直接共有メモリで渡すなど、**メモリを共有できることが前提**の設計
- デプロイ構成をシンプルに保ちたい（単一プロセス・単一バイナリ）

### 代表的な用途

- 数値計算・行列演算（NumPy, SciPy の内部はこれ）
- 画像・音声処理のコーデック
- 暗号処理
- ゲームエンジンの物理演算

---

## マイクロサービスが優れているケース

### 適合条件

- **並列・分散処理がメイン**の設計（スループット重視）
- 各処理を**独立してスケール**させたい
- 言語を**最適な用途に使い分け**たい（Python で ML 推論、Rust で低レイヤー、Go で並行 I/O 処理）
- チームが複数言語を独立して管理・デプロイできる体制がある
- ボトルネックが CPU ではなく **I/O・ネットワーク・DB** にある

### 代表的な構成例

```
Python サービス（ML 推論・オーケストレーション）
    ↓ gRPC / HTTP
Go サービス（高並列 API ゲートウェイ）
    ↓ gRPC
Rust サービス（低レイテンシデータ処理）
```

---

## 今回のベンチマークが示した現実

Windows ネイティブ環境での FFI ベンチマーク結果（実測値）：

| 言語 | 平均スピードアップ | 主な問題点 |
|---|---|---|
| C | 13.6x | ビルド複雑・DLL 管理・メモリ安全性なし |
| C++ | 10.8x | 同上・ABI 互換性問題 |
| Rust | 10.5x | ビルド複雑・borrow checker の学習コスト |
| NumPy | 7.2x | 実態は Python フォールバック（DLL 未実装） |
| Go | 3.2x | CGO オーバーヘッドで期待より低速 |
| Zig | 3.1x | エコシステムが未成熟 |
| Nim | 1.2x | ほぼ改善なし |
| Kotlin | 1.3x | JVM 起動・GC オーバーヘッド |

### 読み取れること

- **Go・Rust を FFI で呼ぶメリットは限定的**。CGO / FFI のオーバーヘッドが大きく、
  マイクロサービス化して gRPC で呼ぶ方がシンプルで保守しやすいケースが多い。
- **C が最速だが、コストが高い**。メモリ安全性がなく、DLL 管理・クロスプラットフォーム
  対応で運用負担が大きい。
- **Nim・Kotlin は FFI 経由では効果が薄い**。これらの言語の強みはFFIではなく
  独立サービスとして動かすことにある。

## 判断フレームワーク

```
ボトルネックはどこか？
├── CPU バウンド（計算集約）
│   ├── レイテンシ < 1ms が必要 かつ 大量データのシリアライズを避けたい
│   │   └── → FFI / 拡張モジュール（C, Rust, Cython）
│   └── それ以外（スループット重視・独立スケール可）
│       └── → マイクロサービス（Rust, Go）
└── I/O バウンド（DB, ネットワーク, ファイル）
    └── → 非同期 Python（asyncio）または Go マイクロサービス
            FFI は効果なし
```

---

## FFI の本質的な立ち位置

> **FFI は「Python の呪縛から抜け出せないときの妥協案」**

設計の自由度がある新規プロジェクトであれば、最初からマイクロサービスで分離する方が
長期的には健全。FFI は以下の制約がある場合に選ぶ手段：

- Python エコシステム（ライブラリ・フレームワーク）への依存が外せない
- 既存コードの大規模な書き直しが困難
- 単一デプロイ単位であることが要件

---

## まとめ：選択基準

| 条件 | 推奨アーキテクチャ |
|---|---|
| 既存 Python コード + 局所的なホットスポット | FFI / 拡張モジュール（C, Rust） |
| マイクロ秒単位のレイテンシ + 大量データ共有 | FFI / 拡張モジュール（C, Rust） |
| 並列処理・スループット重視 | マイクロサービス（Go, Rust） |
| I/O バウンド | asyncio または マイクロサービス |
| 独立スケール・独立デプロイが必要 | マイクロサービス |
| 新規設計で Python 依存なし | マイクロサービス（FFI は不要） |
